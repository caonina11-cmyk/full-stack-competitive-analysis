# Full-Stack Competitive Analysis Skill

一个面向产品经理、产品设计师、设计师和产品负责人的全栈竞品分析 Skill。

它不是用来生成一份“资料堆叠型竞品表”，而是帮助你围绕具体产品目标，形成可用于产品方向、功能取舍、体验路径、增长机制和设计落地的判断。

## 适合做什么

这个 Skill 适合以下任务：

- 多个竞品横向对比
- 单个产品深度拆解
- 基于一个锚点产品自动发现竞品池
- 分析某个赛道里的竞品格局
- 对比官网、注册、转化、核心路径、功能能力或 UX 表达
- 围绕目标做竞品分析，例如提升留存、提升日活、提升转化、功能补齐、MVP 范围判断

## 核心特点

- **目标导向**：如果你给出明确目标，Skill 会先拆出 4-7 个目标拆解关键因素。
- **不同产品不同拆法**：学习产品、社交产品、电商产品、AI 工具、SaaS 产品不会套同一张表。
- **同一产品不同目标也会重拆**：例如 Soul 提升留存、提升付费、提升安全感，会得到不同观察重点。
- **自动竞品池**：如果只给锚点产品和目标，Skill 会自动提出直接竞品、间接/替代方案、潜在竞品和标杆参考。
- **从宏观到微观**：覆盖竞争格局、用户场景、能力取舍、核心路径、流程机制、机会验证和产品与体验决策。
- **证据驱动**：要求区分官方资料、真实体验、行业资料、第三方口碑和信息缺口。

## 安装方式

把这个仓库下载或克隆到你的 skills 目录中，并保持目录名为：

```text
full-stack-competitive-analysis
```

目录结构应类似：

```text
full-stack-competitive-analysis/
  SKILL.md
  references/
    analysis-frameworks.md
    report-templates.md
    research-workflow.md
    ux-product-design-audit.md
    diagrams.md
```

安装后，在支持 Skills 的 Agent/Codex 环境里，提到竞品分析、竞品调研、产品拆解、UX 竞品分析等任务时即可触发。

## 怎么提问

最简单的用法：

```text
请使用 full-stack-competitive-analysis，以 Soul 为核心锚点，围绕“提升陌生人社交产品用户留存”做竞品分析。
```

也可以更开放：

```text
请分析 Notion、飞书文档和 Google Docs 在 AI 文档协作上的竞品差异，并给出产品与体验决策建议。
```

或者：

```text
请以一款在线学习产品为锚点，分析如何提升用户日活。你可以帮我选择合适的竞品池。
```

你不需要提前写死分析维度。Skill 会根据产品类型、分析目标、用户任务和商业模式，自动拆出本次应该重点观察的目标拆解关键因素。

## 输出通常包括

报告结构会根据任务自动调整，但标准深度报告通常会覆盖：

- 核心判断
- 目标拆解关键因素
- 竞品池与选择理由
- 竞争格局
- 用户场景与切换动因
- 产品能力矩阵与 KANO/功能取舍
- 核心路径、流程机制与体验表达
- 机会验证与风险反证
- 产品与体验决策
- 行动建议与 Non-goals
- 来源、可信度与信息缺口

## 和行业机会研究的关系

行业机会研究更偏宏观，用来判断一个行业现在处于什么阶段、有哪些机会方向。

竞品分析更偏微观，用来在已有产品、目标或命题下，研究竞品如何承接用户需求、产品机制和体验路径，并最终转化为可用的产品与体验判断。

一个常见流程是：

```text
行业机会研究
→ 锚定方向或产品命题
→ 使用 full-stack-competitive-analysis 做竞品分析
→ 用户画像
→ 机会点与功能判断
→ 原型与体验设计
```

## 文件说明

- `SKILL.md`：Skill 主入口和核心执行规则。
- `references/research-workflow.md`：信息采集、竞品池选择、目标拆解关键因素和证据规则。
- `references/report-templates.md`：报告阅读路径、模块契约和展示规则。
- `references/analysis-frameworks.md`：商业、战略、用户、功能、UX、增长等分析框架。
- `references/ux-product-design-audit.md`：体验和界面专项分析规则。
- `references/diagrams.md`：需要图表时的表达规则。

## 注意

这个 Skill 会尽量拉高报告下限，但分析质量仍取决于可获取信息、是否能真实体验产品、使用的模型能力和用户给出的任务边界。

如果关键证据不足，报告应该明确标注为初步判断，并说明需要补充哪些信息。
