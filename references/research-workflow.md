# 研究流程与信息采集

本文件用于全栈竞品分析的信息采集、落盘和汇总。目标不是收集最多材料，而是收集足以支撑产品与体验决策的证据。

## 目录

1. 工作目录与落盘
2. 竞品发现与确认
3. 信息源分层与可信度
4. Phase 1 官网与产品入口采集
5. Phase 2 定向扩展搜索
6. Phase 3 文档、报告与深层资料挖掘
7. Phase 4 产品体验证据采集
8. Phase 5 第三方口碑与社区反馈
9. 合并为判断素材

## 1. 工作目录与落盘

在开始采集前创建：

```text
{当前工作目录}/
└── {分析对象或赛道}-资料/
    ├── raw/          # 原始网页、文档、截图说明、抓取内容
    ├── summary/      # 单篇资料摘要
    ├── evidence/     # 产品路径、页面、功能、体验证据
    └── merged.md     # 按分析维度合并后的核心素材
```

落盘规则：

- 每次抓取后立即保存到 `raw/`，文件开头记录来源 URL、抓取日期、资料类型。
- 每个 `raw/` 文件提炼成对应 `summary/`，保留关键发现、数据、来源可信度和可用于判断的证据。
- 产品体验、路径、页面结构、截图观察、试用记录保存到 `evidence/`。
- 最后把所有摘要合并为 `merged.md`，按“行业、用户、商业、产品、功能、体验、增长、战略信号、产品与体验决策线索”组织。

不要把大量 raw 内容堆在上下文里。按文件读取、提炼、保存，再进入下一份。

## 2. 竞品发现与确认

竞品选择本身就是分析判断的一部分。不要只选择“最像自己”的产品；要根据分析目标和产品阶段判断哪些对象能帮助回答问题。

选择竞品前先校准：

- 为什么做这次竞品分析：方向判断、功能补齐、体验优化、增长转化、商业定价、单点验证，还是推荐率/活跃度/留存等具体目标。
- 我方或分析对象处于什么阶段：探索期、增长期、成熟期、转型期；阶段不同，直接对标、标杆学习和避开战场的权重不同。
- 当前最大问题是什么：竞争力不足、转化低、留存差、功能不完整、体验难用、价值表达不清，还是新机会不确定。
- 需要哪些类型的参照物：直接竞品用于看牌桌标准，间接/替代方案用于看真实任务竞争，头部/标杆对象用于看用户预期和趋势，潜在竞品用于看未来威胁。

当分析对象是中小产品、新产品或新进入者时，必须特别判断它是否正在和资源、渠道、生态能力明显更强的头部产品正面竞争。若是，报告中要寻找可避开的细分场景、垂直人群、工作流断点、体验差异化或商业模式差异；不要默认建议正面复制头部产品。

### 用户已指定竞品

优先分析用户确认的对象，但不要把它们视为绝对边界。用户可能只给 1-2 个熟悉产品，真实判断仍需要少量直接竞品、替代方案、潜在竞品或标杆参考来校准。

允许补充竞品的情况：

- 用户给出的名单不足以回答本次目标，例如只给直接竞品，但目标涉及留存、转化、商业化或体验范式，需要替代方案或标杆参考。
- 研究发现某个产品明显影响用户选择、任务替代、行业预期或未来竞争格局。
- 用户名单里缺少头部/标杆对象，导致无法判断行业标准、用户预期或能力边界。
- 用户名单里缺少间接/替代方案，导致看不到真实任务竞争。

补充规则：

- 新增对象必须少量、必要，不能为了显得全面而扩充。
- 每个新增对象必须说明“为什么加入”“竞争关系是什么”“它帮助回答本次哪个问题”。
- 如果新增对象会显著改变分析范围、行业边界或报告长度，先请用户确认。
- 如果只是作为背景参照或标杆参考，可以直接纳入，但必须和用户指定竞品分层区分。

### 用户要求发现竞品

先轻量研究目标公司/产品，回答：

- 它解决什么问题
- 服务谁
- 所属赛道和细分市场
- 主要价值主张和产品形态

然后提出 3-5 个候选竞品，按类型标注：

- 直接竞品：同品类、同目标用户、同核心任务
- 间接竞品：不同产品形态，但争夺同一用户任务、预算、时间或使用场景
- 替代方案：用户不用该产品时会采用的人工流程、旧工具、组合方案或外包服务
- 潜在竞品：当前不是直接竞品，但可能凭技术、渠道、平台、生态或资本能力切入
- 头部/标杆对象：不一定适合正面对标，但可用于观察趋势、用户预期、服务标准、体验范式或增长打法

如果用户只给锚点产品和分析目标、没有给竞品名单，可以先生成初始竞品池并直接进入标准深度分析；不要因为缺少名单而阻塞。只有当候选过多、赛道边界不清、或不同竞品池会导向完全不同结论时，才先让用户确认。

候选名单不要只列名字。每个候选对象至少说明：

- 为什么选它：它能帮助回答本次哪个分析目标。
- 竞争关系：直接、间接、替代、潜在、头部/标杆。
- 对决策的意义：用于学习、避开、补齐、验证机会，还是判断未来威胁。

如果用户只给行业或方向，没有给锚点产品或具体竞品，先做快速竞品池扫描，并让用户确认重点对象后再进入完整分析。

## 2.1 目标拆解关键因素

具体目标会改变分析口径。执行采集前，先把目标拆成适合该产品类型和本次目标的 4-7 个候选关键因素假设；不要照搬其他行业示例，也不要把同一产品其他目标的拆解直接复用过来。采集和竞品观察之后，再把这些候选项校准为正式报告里的目标拆解关键因素。这些关键因素同时是后续真实体验竞品时的竞品观察维度。

拆解步骤：

1. 定义目标：这次到底要判断活跃、留存、转化、推荐、功能补齐、体验优化、市场进入还是 MVP 范围。
2. 识别关键行为：该产品的价值由什么行为产生，例如学习、创作、协作、交易、搜索、生成、分享、复访、付费或交付。
3. 拆 4-7 个候选关键因素：每个候选项都要能解释目标，不要为了完整而堆指标。
4. 给候选关键因素配证据：为每个候选项记录需要的来源类型，例如官方功能、产品路径、用户口碑、行业数据、价格/套餐或战略信号。
5. 做证据校准和权重判断：哪些关键因素最可能改变产品与体验决策，哪些只是背景参考，哪些应降级为待验证假设。

报告要求：如果用户给出明确目标，标准深度报告需要显式呈现证据校准后的目标拆解关键因素。它可以独立成节，也可以放在核心判断、竞争格局或用户场景开头；但不能只在后文机会表或流程表中隐含出现。读者必须看得出“为什么后文按这些关键因素分析”，学生或执行者也应能带着这些关键因素去真实体验竞品。若某些关键因素仍缺证据，必须标注为待验证假设。

示例：

| 产品类型 | 目标 | 可能关键因素 |
|---|---|---|
| 线上学习平台 | 用户活跃度 | 内容供给、学习路径、作业反馈、社群互动、运营活动、激励机制 |
| AI 原型工具 | 用户活跃度 | 生成成功率、编辑控制感、项目沉淀、模板复用、协作交接、返修路径 |
| 在线文档/知识库 | 留存与协作 | 权限、评论通知、搜索、模板、知识沉淀、工作流集成 |
| 电商/交易产品 | 转化与复购 | 搜索推荐、详情页信息、信任证据、促销机制、履约体验、会员复购 |
| 社区产品 | 活跃与创作 | 内容分发、互动反馈、关系链、身份体系、创作激励、治理机制 |

示例只用于说明拆解方法。正式报告中的关键因素必须根据本次对象、目标和证据重新生成。

关键因素输出可以是表格、路径图或段落，但必须说明它告诉我们什么、为什么和本次目标相关、对后续竞品选择、真实体验或产品决策有什么影响。

## 2.2 流程内机制识别

线上产品、工具产品、社区产品、交易产品、学习产品、社交产品和内容产品，不能只采集“功能清单”。必须把关键用户流程拆成阶段，并记录每个阶段竞品使用了什么机制推动用户继续。

流程内机制是增强层，不是报告主线。采集时仍要先保留竞争格局、用户场景、能力取舍和核心路径证据；机制素材用于解释“功能如何在流程里发挥作用”，并帮助后续机会验证和功能假设生成。

机制识别步骤：

1. 拆关键流程：入口、理解价值、完成首个动作、获得反馈、继续使用、沉淀资产、复访/召回、付费/转化、退出/安全。
2. 定义用户阻力：用户在这一阶段为什么会停下、流失、犹豫、误解、焦虑或无感。
3. 记录竞品机制：竞品用什么功能、内容、交互、推荐、激励、AI、社交关系、安全提示、商业规则或运营动作解决这个阻力。
4. 判断机制作用：这个机制影响的是激活、转化、留存、复购、创作、信任、效率、关系沉淀还是商业化。
5. 转成产品判断：该机制是基础能力、期望能力、兴奋能力，还是当前不该跟的过度建设。

常见流程内机制：

| 产品类型 | 常见流程阶段 | 典型机制 |
|---|---|---|
| 陌生人/兴趣社交 | 进入、发现人、破冰、继续聊天、沉淀关系、复访 | 人格测试、兴趣标签、语音匹配、AI 破冰、共同话题、关系进度、互动回顾、安全提醒 |
| 学习产品 | 设定目标、开始学习、练习反馈、连续学习、复习、社群互助 | 每日任务、学习路径、错题复习、即时反馈、连续打卡、排行榜、老师点评 |
| AI 工具 | 输入意图、生成结果、编辑控制、复用沉淀、协作交接 | 模板、示例提示、生成预览、局部编辑、版本历史、项目资产、协作分享 |
| 电商/交易 | 搜索、详情理解、信任建立、下单、履约、复购 | 推荐排序、评价证据、价格锚点、保障承诺、物流反馈、会员权益 |
| 内容/社区 | 内容发现、参与互动、创作发布、获得反馈、身份成长 | 分发机制、话题挑战、评论反馈、创作者激励、身份等级、治理规则 |

采集时优先保存为“阶段-阻力-机制-证据-判断”结构：

| 流程阶段 | 用户阻力 | 竞品机制 | 证据类型 | 对目标的影响 | 我方判断 |
|---|---|---|---|---|---|
| | | | 官方/实测/评论/推断 | | |

如果没有实测，只能写“基于官方资料/第三方材料推断”，不能写成完整体验结论。

## 3. 信息源分层与可信度

全栈竞品分析不能只看官网。不同信息源支撑不同判断，报告中要标注来源类型、日期和可信度。

| 层级 | 信息源 | 主要用途 | 适合支撑的判断 |
|---|---|---|---|
| L1 官方产品源 | 官网、产品页、定价页、帮助中心、开发者文档、更新日志、官方博客、客户案例 | 获取官方定位、产品结构、功能清单、定价、CTA 和最新发布 | 产品定位、功能体系、核心路径、表达判断 |
| L2 真实体验源 | 公开试用、Demo、产品内路径、控制台、移动端、截图、录屏、可访问性和性能观察 | 验证实际用户旅程，而不是只信宣传文案 | 体验判断、交互摩擦、首次价值、转化路径 |
| L3 商业与公司源 | 年报、财报、投资者关系、融资新闻、公司公告、并购信息、客户合同 | 判断商业模式、收入压力、增长逻辑、战略方向 | 商业模式、增长判断、战略风险 |
| L4 权威行业报告 | Gartner、Forrester、IDC、艾瑞、QuestMobile、灼识咨询、易观、券商研报、36氪研究院、行业协会报告 | 判断市场规模、行业趋势、竞争格局、买方变化 | 行业环境、竞争格局、方向判断 |
| L5 战略信号源 | 专利、招聘、领导层变化、组织调整、生态合作、开放平台、API、伙伴计划 | 判断公司真实押注方向，而不只看它怎么宣传 | 战略信号、未来路线、组织能力 |
| L6 第三方口碑源 | 用户评价、媒体评测、社区讨论、应用商店评论、社媒反馈、客户访谈/案例 | 判断真实痛点、口碑、未满足需求和使用摩擦 | 用户判断、体验判断、机会地图 |

可信度优先级通常为：官方披露/监管披露/财报 > 权威报告/行业数据库 > 官方产品资料 > 实际体验证据 > 媒体报道 > 社区口碑。实际使用时不要机械排序：体验问题优先看真实体验证据，市场规模优先看权威报告，战略押注优先看招聘/专利/组织变化。

使用要求：

- 关键事实必须标注来源；推断必须写明依据。
- 关键数字必须逐条标注来源、时间和可信度，不要只在文末列“来源集合”。
- 标准深度报告至少覆盖 L1 官方产品源、L2 真实体验源、L4 权威行业报告/市场资料、L6 第三方口碑源中的 3 类；不足 3 类时，必须降级为“初步判断版”。
- 如果关键判断主要来自官网或营销页，只能作为初步判断，不能写成完整结论。
- 行业报告优先选择最近 12-24 个月；AI、SaaS、协作工具等变化快的领域优先最近 6-12 个月。
- 权威报告可以支撑宏观趋势和市场判断，但不能替代产品体验走查。
- 社区口碑只能作为问题线索，不能单独作为重大结论依据，除非有多源交叉验证。
- 不同来源冲突时，保留冲突并说明可能原因，不强行合并成单一结论。
- 付费报告、摘要页或转述资料要标注可见范围，避免把二手摘录当成完整报告。
- 写“未覆盖”前必须说明原因：没有执行采集时写“本次未采集”；执行过检索但无有效结果时写“已检索但未获得有效材料”；受登录、付费墙、权限或地域限制时写“可见范围受限”。不要把未采集写成资料不存在。
- 正式试跑或验收 skill 时，不能只做报告结构 dry run。至少要实际尝试采集 L1、L2、L4、L6 中 3 类信息源，记录可获取性和缺口，才能判断信息源规则是否有效。

关键判断的最低证据要求：

| 判断类型 | 最低证据要求 | 证据不足时写法 |
|---|---|---|
| 产品定位/功能体系 | L1 官方产品源，必要时补帮助文档或更新日志 | “基于官方资料判断” |
| 核心路径/体验表达 | L2 实测优先；无法实测时用官方演示、帮助文档、第三方截图/评测 | “基于 {证据类型} 观察/推断，需实测复核” |
| 市场趋势/赛道格局 | L4 权威报告、研报、行业数据库或可信媒体 | “缺少权威市场数据，暂作趋势假设” |
| 用户痛点/切换动因 | L6 用户评价、社区讨论、客户案例、访谈或第三方评测 | “缺少用户侧证据，暂作推断” |
| 战略方向/未来押注 | L5 招聘、专利、组织变化、生态合作、API、更新日志 | “战略信号不足，不能作为强结论” |

## 4. Phase 1 官网与产品入口采集

先从官方信息开始，建立第一手认知。

必采信息：

- 官网首页、产品页、解决方案页、定价页、关于我们页
- 导航结构、底部链接、核心 CTA、注册/试用/联系销售入口
- 产品矩阵、核心功能、目标客户、应用场景、客户案例
- 帮助中心、文档中心、更新日志、博客、新闻动态

优先输出：

- 官网链接地图
- 产品矩阵初稿
- 核心用户路径入口
- 已知功能清单
- 信息空白点

如果能进入公开产品、试用版、演示环境或控制台，应优先记录真实路径，而不是只依赖官网文案。

## 5. Phase 2 定向扩展搜索

根据 Phase 1 的空白点定向搜索，避免漫无目的泛搜。

常见搜索方向：

- `{产品名} 定价/收费/商业模式`
- `{产品名} 功能/教程/使用手册/帮助文档`
- `{产品名} 用户评价/评测/体验`
- `{公司名} 融资/营收/市场份额/客户案例`
- `{行业} 市场规模/竞争格局/趋势`
- `{产品名} AI/自动化/开放平台/生态`
- `{公司名} 招聘/岗位/产品经理/设计师/AI`
- `{行业} Gartner/Forrester/IDC report`
- `{行业} 艾瑞/QuestMobile/易观/灼识/券商研报`
- `{产品名} review/pricing alternatives`

每条重要信息要标注来源和日期。优先使用最近 12-24 个月资料；变化很快的赛道优先最近 6-12 个月。

标准深度报告的最小采集尝试：

- L1：每个关键竞品至少官网/产品页/定价页/帮助中心中的 2 类。
- L2：至少选择 1-2 条核心路径做实测、官方 demo/帮助文档还原，或第三方 walkthrough/截图复核。
- L4：至少检索 3 组行业/市场关键词，优先找权威报告、研报、数据库摘要或可信媒体数据。
- L6：至少检索每个关键竞品的用户评价、评测、替代方案讨论或社区反馈中的 1-2 类。

如果时间或权限不允许完成完整采集，报告应标为“初步判断版”，并写清楚当前证据覆盖范围：哪些来源已经尝试过，哪些只是本次未采集，哪些受可见范围限制。

## 6. Phase 3 文档、报告与深层资料挖掘

重点挖掘官网不会主动公开的资料：

| 文档类型 | 价值 |
|---|---|
| 使用手册/帮助文档 | 功能深度、真实操作流程 |
| 招投标文件 | 真实定价、技术参数、交付标准 |
| 白皮书/方案书 | 技术架构、行业定位、价值主张 |
| 培训材料/PPT | 销售话术、客户场景、产品卖点 |
| 年报/财报/投资者材料 | 收入结构、战略重点、风险 |
| 权威行业报告/研报 | 市场规模、竞争格局、趋势、买方变化 |
| 专利/招聘/领导层变化 | 战略信号和真实投入方向 |

搜索格式示例：

```text
"{产品名} 使用手册 filetype:pdf"
"{产品名} 招标 采购 filetype:pdf"
"{公司名} annual report"
"{公司名} patent AI"
"{公司名} product designer jobs"
"{行业} market report filetype:pdf"
"{行业} 研报 filetype:pdf"
"{行业} 白皮书 filetype:pdf"
```

无法解析或来源较弱的资料要标注，不要强行使用。

## 7. Phase 4 产品体验证据采集

做全栈竞品分析时，体验证据和商业/功能证据同等重要。

至少观察：

- 首页到核心 CTA 的路径
- 注册/登录/试用/预约演示路径
- 首次使用或 onboarding
- 核心任务路径
- 付费/升级/联系销售路径
- 帮助、空状态、错误状态、引导、反馈
- 移动端和响应式体验

证据来源分级：

| 类型 | 说明 | 报告写法 |
|---|---|---|
| 实测 | 实际注册、试用、进入产品、完成路径 | “实测观察” |
| 官方演示/官网 | 官网截图、产品视频、官方 demo、营销页路径 | “基于官方演示/官网观察” |
| 帮助文档 | 帮助中心、使用手册、开发者文档 | “基于帮助文档推断路径” |
| 第三方截图/评测 | 媒体评测、用户教程、社区截图、视频 walkthrough | “基于第三方材料观察，需复核” |
| 推断 | 只从产品定位、功能说明或价格页推导 | “推断，未实测” |

核心路径、优缺点和界面表达必须尽量来自 L2 真实体验源。无法实测时，可以使用官方演示、帮助文档、第三方截图补充，但必须标注来源类型，不能把推断写成实测。

记录格式：

```markdown
## {竞品名} - {路径名}

- 入口：
- 用户目标：
- 步骤数：
- 关键页面：
- 摩擦点：
- 亮点：
- 可能的商业意图：
- 对我方的产品设计启示：
- 证据来源：
```

## 8. Phase 5 第三方口碑与社区反馈

第三方口碑用于发现真实摩擦和未满足需求，但必须谨慎使用。

建议采集：

- 应用商店评论和评分变化
- G2、Capterra、Product Hunt、知乎、少数派、即刻、Reddit、X 等社区讨论
- 媒体评测、用户教程、竞品替代清单
- 客户案例中的真实使用场景和引用

记录格式：

```markdown
## {竞品名} - 第三方口碑摘要

- 来源：
- 时间：
- 用户类型：
- 高频正面反馈：
- 高频负面反馈：
- 可能反映的体验/产品问题：
- 是否有其他来源交叉验证：
- 对产品与体验决策的意义：
```

不要把孤立吐槽写成结论。至少要判断它是个体抱怨、特定版本问题，还是多个来源反复出现的结构性问题。

## 9. 合并为判断素材

`merged.md` 不按来源堆叠，而按判断维度组织：

1. 行业与竞争环境
2. 用户、买方、场景和任务
3. 商业模式和产品定位
4. 产品矩阵和功能体系
5. 核心路径和体验证据
6. 界面表达、文案、视觉、内容策略
7. 增长、转化、留存和运营机制
8. 战略信号：专利、招聘、组织、资本、产品更新
9. 权威报告与行业数据要点
10. 第三方口碑与用户反馈
11. 产品与体验决策线索
12. 信息缺口与可信度

写报告时优先读取 `merged.md`，必要时回到 raw、summary、evidence 查证。

合并素材时同步准备默认判断层：

- **产品能力对比矩阵**：整理关键能力维度、各竞品支持情况、证据类型和对我方优先级的影响。
- **KANO / 功能取舍判断**：初步区分基础型、期望型、兴奋型、暂不跟进/过度建设能力。
- **流程内机制判断**：整理关键流程阶段、用户阻力、竞品机制、机制作用、证据类型和可转化的功能机会。
- **用户痛点与切换路径**：整理用户评论、社区讨论、客户案例或可标注的推断画像。
- **机会优先级**：整理多个机会及其证据强度、用户痛感、差异化空间、实现难度和战略适配度。
- **市场趋势与战略信号**：整理可用的行业数据、政策/监管、融资、采用率、招聘、专利、产品更新等线索。
- **Non-goals**：当报告用于新产品切入或 MVP 定义时，整理足够证据支持“不做什么”。

对按需增强层只做轻量标记：哪些素材可能改变关键判断，哪些素材证据不足。不要为每个增强层都建完整清单；素材不足时不要硬写，在信息缺口里说明还需要补什么证据。
