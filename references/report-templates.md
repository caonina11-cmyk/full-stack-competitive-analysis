# 报告模板

模板的目标是让报告更好读，而不是展示所有分析痕迹。标准深度报告的第一优先级是**判断链**，不是模块目录。默认叙述顺序是：

```text
先决策 → 再证据 → 再机会 → 再行动
```

## 目录

1. 阅读原则
2. 判断链与后台任务清单
3. 模块分析契约
4. 多品竞品分析报告模板（标准深度）
5. 单品、赛道、快速请求适配规则
6. 按需增强模块

## 1. 阅读原则

- **不要为分析而分析**：每一节都要回答“这对产品方向、用户场景、功能策略、体验路径或增长动作有什么用”。
- **框架隐身**：PEST、KANO、商业模式画布、UX 五层等用于内部组织判断，不作为正文方法论章节。
- **判断链优先**：先组织“核心答案 → 竞争格局/关键张力 → 用户场景 → 能力取舍 → 核心路径/流程机制 → 机会/反证 → 产品与体验决策”，再决定用哪些标题、表格和图示。不要先按模块目录填内容。流程机制是从能力取舍走向机会验证的桥梁，不能替代竞争格局、用户场景和能力取舍。
- **先判断后展开**：先给核心结论和推荐方向，再展开证据。
- **先设计阅读路径**：正式写报告前，内部形成 report spine：核心答案、关键证据、主要张力、机会/风险、决策和下一步。展示形式跟随这条阅读路径。
- **段落级判断责任**：每个主要段落、图表、表格和清单都必须支撑一个明确判断。不要放“看起来完整但不改变判断”的信息块。
- **模块是后台任务，不是前台目录**：默认报告围绕“核心决策卡片 + 11 类后台分析任务”做完整性检查；最终正文可以合并、重命名或调整顺序，主体章节优先使用“模块定位：本节核心判断”的复合标题。
- **模块少，信息不薄**：压缩的是重复模块和方法论展示，不是压缩关键事实、观察和推理。每个必要模块都要保留足够信息来支撑判断。
- **标准深度不是摘要集合**：标准深度报告要让读者看到事实、产品细节、用户场景、证据和推理如何形成判断。每个主体模块不能只有一段概括或一张表。
- **混合展示**：不要全用表格。表格用于对比和归纳，短段落用于解释判断，案例/观察用于保留细节，路径图用于体验，机会地图用于取舍。
- **表格不能替代分析**：复杂判断不要只填表。表格前要有“这一节说明什么”，表格后要有“所以我们怎么判断”。
- **展示方式服务内容**：模板提供信息架构，不是固定版式。每个模块根据内容属性选择段落、表格、画像、路径图、定位图、机会地图、决策矩阵或来源表。
- **每段都有工作**：每个主要段落或图表都要支撑一个明确 claim，并配套证据、解释和 implication。不要放没有承担判断任务的表格、图表或段落。
- **来源透明**：关键数字必须逐条标注来源、时间和可信度；体验判断必须标注实测、官网/演示、帮助文档、第三方材料或推断。

写作时内部使用“三层组织”，但不要把这些层级标题原样写进最终报告：

```text
先给本节结论：让读者知道这一节在说明什么。
再展开业务细节：用分层、画像、路径、案例、数据、用户声音解释为什么。
最后形成判断：用表格、路径图、机会图或短结论做归纳。
```

最终报告不要出现“判断摘要 / 证据展开 / 结构化呈现”这类模板提示词。应该使用真实业务标题，例如“市场分层”“关键竞品画像”“竞争张力”“用户切换路径”“能力空白”“设计启示”等。

不要把报告写成只有表格的资料表。表格适合装“横向对比”，不适合承载“为什么重要、背后意图、对我方意味着什么”。

默认结构是后台任务清单，不是强制可见目录。最终报告应该优先使用“模块定位：本节核心判断”的复合标题，例如“竞争格局：Notion 正在把文档变成 AI 工作空间”“用户场景：中小团队真正缺的是开箱即用的工作对象”“战略取舍：不要正面复制全能工作空间”。只要模块契约被完成，就可以合并、重排或改名。

正式写作前内部组织 report spine：

```text
用户要做的决策
→ 当前最可靠的答案
→ 支撑答案的 3-5 个关键发现
→ 每个发现的证据和不确定性
→ 机会、风险与反证
→ 产品与体验决策
→ 下一步验证或行动
```

展示方式选择规则：

| 内容属性 | 优先展示方式 | 不适合的方式 | 使用要求 |
|---|---|---|---|
| 横向比较多个竞品的同一维度 | 对比表、能力矩阵 | 长篇散文式并列描述 | 表格后必须写出关键差异和对我方的含义 |
| 解释单个竞品的产品逻辑 | 画像段落、结构化小节 | 只有一行表格 | 写清用户、任务、产品结构、增长入口、优势、限制 |
| 展示用户完成任务的过程 | 路径图、任务矩阵、步骤观察 | 只写“体验好/差” | 标注证据类型，并说明摩擦点和背后意图 |
| 说明市场位置和竞争关系 | 分层地图、二维定位图、生态关系图 | 纯功能表 | 坐标轴或分类标准必须来自分析对象，不套通用轴 |
| 展开用户场景和切换动因 | 场景叙事、用户声音、切换路径表 | 只有用户类型列表 | 写清任务压力、现有替代方案、真实损失和机会 |
| 判断机会优先级 | 机会地图、机会/风险表、P0/P1/P2 | 只列建议清单 | 同时写机会依据和不成立条件 |
| 沉淀最终判断 | 决策矩阵、综合判断段落 | 只重复前文摘要 | 把方向、用户、功能、体验、表达、增长和优先级串起来 |
| 说明证据和可信度 | 来源覆盖表、推断边界清单 | 把来源混在正文里不标注 | 区分已采集、已检索未获得、可见范围受限和证据冲突 |

一份专业报告通常会混合使用多种呈现方式。除来源表和少量矩阵外，不要连续多个主体模块都只用表格；如果某节必须用表格承载信息，表格前后必须有解释和判断。

图表、表格和卡片的使用标准：

- 每个图表或表格必须有相邻解释，说明它支持什么判断、如何阅读、对我方意味着什么。
- 图表标题和表格标题只负责命名，不承担结论；结论要写在相邻正文里。
- 表格适合精确比较和归纳，不适合承载长推理。长推理用段落、场景叙事、路径观察或机会论证。
- 图表、表格和清单只是证据表达方式，不是报告主体；如果它们不能支撑明确判断，就删掉、合并或改成段落。
- 同一信息不要同时用表格和图表重复展示，除非两者承担不同阅读任务。
- 信息不足时跳过图表，不用占位图、空表或为了好看而画图。
- 多个图表连续出现时，要确认它们分别回答不同问题，不能默认每节都放一个图。

段落、图表、表格的最小验收标准：

```text
这一块支撑的判断是什么？
它使用的证据是什么？
它对产品方向、功能策略、体验路径、表达方式或优先级意味着什么？
```

如果回答不出来，就不要保留这一块。

## 2. 判断链与后台任务清单

默认报告通常需要先形成一条判断链，再检查“核心决策卡片 + 11 类后台分析任务”是否完成。

### 2.1 推荐判断链

标准深度报告优先采用这种正文骨架：

```text
核心决策卡片
→ 这个赛道/竞品集合真正的竞争张力是什么
→ 用户在什么场景下为什么会切换或犹豫
→ 哪些产品能力已经标配化，哪些能力才影响选择
→ 关键竞品分别定义了怎样的核心路径和产品意图
→ 哪些机会值得做，为什么可能成立，为什么可能不成立
→ 我方产品与体验决策、行动建议和 Non-goals
→ 来源、可信度与信息缺口
```

可见标题要尽量采用“模块定位：本节核心判断”的复合标题，例如：

- “用户场景：真实需求不是帮我画图，而是帮我更快做判断”
- “能力取舍：生成能力正在标配化，差异来自生成之后”
- “核心路径：不同产品正在定义不同的从想法到落地”
- “机会验证：这个机会成立在中间层，失效也可能来自中间层被平台吃掉”

不要默认写成：

- “用户痛点、场景与切换动因”
- “产品能力对比矩阵”
- “核心路径、流程机制与体验表达”

也不要完全去掉模块定位，只写散文化判断，让读者看不出这一节承担的是竞争、用户、能力、路径还是机会任务。模块名可以调整成更自然的短词，例如“竞争格局”“用户场景”“能力取舍”“核心路径”“机会验证”。

### 2.2 后台任务清单

写作时必须完成以下“核心决策卡片 + 11 类后台分析任务”，但它们可以融入判断链，不必逐项成为章节：

```text
0. 核心决策卡片
1. 竞品格局与画像
2. 用户痛点、场景与切换动因
3. 市场趋势与战略信号（轻量默认，证据足够时展开）
4. 商业模式、产品定位与增长逻辑
5. 产品能力对比矩阵
6. KANO / 功能取舍判断
7. 核心路径、流程机制与体验表达
8. 机会优先级与风险反证
9. 产品与体验决策
10. 行动建议与 Non-goals（新产品切入时默认）
11. 来源、可信度与信息缺口
```

默认判断层要求：

- **产品能力对比矩阵默认出现**：多品竞品分析必须有核心能力矩阵，用于呈现关键能力差异、证据边界和对我方优先级的影响。证据不足时做“初步能力矩阵”，并标注基于官方资料/需实测复核。
- **KANO / 功能取舍判断默认出现**：不必在正文解释 KANO 方法论，但必须把能力分成基础型、期望型、兴奋型、暂不跟进/过度建设，并说明我方 MVP、体验优先级和差异化取舍。
- **用户痛点与切换路径默认出现**：必须回答用户为什么选择、放弃或切换竞品。没有真实用户证据时，必须标注为推断画像或待验证假设。
- **流程内机制默认观察但不抢主线**：线上产品必须把关键路径拆成阶段，说明用户在每个阶段的阻力、竞品使用的推动机制、该机制影响的目标，以及是否能转化为我方可落地功能。机制观察必须融入“核心路径”“能力取舍”或“机会验证”，不应替代竞争格局、用户场景和能力取舍，也不应默认成为报告前半部分的主章节。
- **机会优先级默认出现**：报告必须帮助判断先做什么、后做什么、不要做什么。机会很多时使用评分表；机会较少时用 P0/P1/P2 或机会-风险段落表达。
- **市场趋势与战略信号轻量默认**：至少说明当前赛道变化、技术/商业/监管趋势或战略信号；只有有可靠数据时才展开市场规模、增长率、融资和采用率。
- **Non-goals 在新产品切入类报告中默认出现**：当报告用于判断新产品方向、MVP 或路线图时，必须说明不建议做什么；纯竞品复盘或资料梳理时可省略。

按需增强模块只在必要时加入：

- 详细 Non-goals：适合需要展开说明放弃方向、触发条件和重新评估条件的战略取舍。
- 行动路线图：适合立项、改版、季度规划。
- 详细 SWOT：适合需要传统战略分析输出。
- 详细行业报告摘要：适合有权威报告和市场数据时。
- 完整关键数据可信度表：适合数字密集型报告。
- 详细战略信号：适合需要分析招聘、专利、组织、资本动作时。

## 2.3 按需增强层轻量原则

按需增强层不是 checklist，也不是固定目录。执行时只做一次轻量判断：**这类信息会不会改变读者的产品与体验决策？** 如果会，而且证据够，就展开；如果不会，或证据不足，就不展开。

轻量原则：

- **默认层先完成**：先完成“核心决策卡片 + 11 类后台分析任务”，不为了增强层打断主报告阅读路径。
- **会改变结论才展开**：详细能力矩阵、详细市场数据、用户感知标签、详细机会评分、SWOT 等，只在它们会影响方向、优先级、MVP 或 Non-goals 时展开。
- **证据够才展开**：市场数据要有来源、时间和口径；用户声音要区分真实评论和推断画像；评分要有评分维度和依据。
- **展开后必须服务判断**：增强层不能只是资料表，必须说明它支持什么判断、改变什么取舍、暴露什么风险。
- **快速任务默认不展开**：快速判断简报只保留必要增强；缺口写到“来源、可信度与信息缺口”，不要为了完整而拉长报告。

常见增强层包括：详细能力矩阵、详细市场数据与趋势、用户内心独白/感知标签、详细机会评分、三类证据交叉验证、详细 Non-goals、SWOT、行动路线图、详细战略信号。它们可以独立成节，也可以融入默认模块。

## 2.4 不同竞品分析目的的权重

不同目的不使用不同模板，只调整报告重点。所有报告仍然遵守判断链、证据边界、段落级判断责任和“模块定位：本节核心判断”的标题规则。

| 目的类型 | 默认输出形态 | 必须加深 | 可以轻量 |
|---|---|---|---|
| 方向型 | 标准深度报告 | 竞争格局、商业/战略信号、用户场景、能力取舍、机会反证、Non-goals | 微观界面细节按需 |
| 功能型 | 功能专项分析或标准报告中的功能加权版 | 功能场景、能力矩阵、KANO、核心路径、MVP 边界 | 市场规模、宏观趋势 |
| 体验型 | 体验专项分析 | 核心路径、交互摩擦、页面结构、状态反馈、设计启示 | 完整商业模式、行业规模 |
| 表达型 | 表达/官网/页面专项分析 | 信息架构、价值主张、文案、视觉层级、CTA、信任证据 | 深度功能矩阵 |
| 增长/商业型 | 商业与增长专项分析 | 定价、套餐、增长入口、转化路径、留存机制、商业目标对体验的影响 | 微观交互细节 |
| 单点验证型 | 快速判断简报 | 明确结论、2-4 个证据、是否标配、我方建议、信息缺口 | 完整纵深报告 |

## 2.5 目标、竞品选择与目标拆解关键因素

当用户给出明确分析目标时，报告需要让读者看见“为什么这样分析”，但不要单独写成方法论章节。最自然的位置通常在核心判断、竞争格局或用户场景的开头。

必须完成三件事：

- **说明目标如何影响分析重点**：例如本次是看新产品切口、功能补齐、体验优化、活跃度、留存、转化还是推荐率。
- **说明竞品为什么被选中**：直接竞品、间接竞品、替代方案、潜在竞品、头部/标杆对象分别承担什么判断任务。
- **说明目标拆解关键因素为什么适配本行业和本目标**：不要照搬其他行业、其他产品或同一产品其他目标的维度。关键因素必须来自本产品的核心用户行为、商业模式、价值实现路径、本次分析目标和已采集到的竞品/用户/市场证据。

写法要求：

- 目标和关键因素可以合并进“核心判断”“竞争格局”“能力取舍”等章节，不必新增独立章节。
- 当用户明确给出目标时，关键因素拆解必须对读者可见，尤其是留存、活跃、转化、推荐率、MVP 范围、功能补齐这类会直接决定分析口径的目标。可以不叫“目标拆解”，但必须写清目标如何拆成 4-7 个经过证据校准、适配本行业和本目标的关键因素，以及这些关键因素如何影响后续竞品选择、真实体验和机会判断。
- 如果使用关键因素表，表格前后必须说明它如何指导后续竞品观察、对比和决策。
- 如果用户没有说明目标，默认按综合标准分析生成核心观察重点，不要为了展示方法论而写“本次分析目标识别”。
- 如果关键因素来自推断而非证据，标注为“关键因素假设”，并在信息缺口中说明需要怎样补证据。
- 报告开头前的工作说明或状态更新，不要提前宣布最终关键因素；最多说明“将先形成候选关键因素假设，并通过资料和竞品证据校准”。

可用表达示例：

```markdown
本次报告不是只比较功能多寡，而是判断 {目标}。因此后文会把竞品放在 {直接竞品/替代方案/标杆对象} 三类关系里看，并优先观察 {4-7 个经过证据校准、适配本行业和本目标的关键因素}。这些关键因素来自 {用户任务/商业模式/核心路径/竞品证据}，不是通用指标表；后续真实体验竞品时也应围绕它们采集证据。
```

不要写成：

- “以下是本次使用的方法论。”
- “所有产品都从课程内容、运营活动、社交模块、激励玩法分析。”
- “因为模板要求，所以选择这些指标。”

## 3. 模块分析契约

标准深度报告不靠字数或句数定义深度，而靠每个模块是否完成了必要的分析任务。写作时每个主体模块都要满足“问题、证据、分析动作、展示方式、完成标准”五项契约。句数、表格数量和小标题数量只作为表达手段，不作为质量本身。

### 3.1 核心决策卡片

| 契约项 | 要求 |
|---|---|
| 要回答的问题 | 这份报告最重要的结论是什么？现在应该优先做什么、避免什么、验证什么？ |
| 必须使用的证据 | 后文最关键的竞品、用户、市场、体验和风险证据。 |
| 必须完成的分析动作 | 把分散发现压缩成推荐方向、优先级、关键依据、主要风险和下一步验证。 |
| 推荐展示方式 | 决策卡片或短表格。 |
| 完成标准 | 读者即使只读这一节，也能知道当前最重要的判断，但不会误以为已经看到全部证据。 |

### 3.2 竞品格局与画像

| 契约项 | 要求 |
|---|---|
| 要回答的问题 | 这个市场按什么逻辑竞争？哪些是直接竞品、替代方案、标杆对象？我方不该硬拼什么，应该观察或切入哪里？ |
| 必须使用的证据 | 官方定位、产品结构、目标用户、定价/增长入口、客户案例、行业/媒体资料。 |
| 必须完成的分析动作 | 竞品分层、关键竞品画像、竞争张力提炼、格局判断。 |
| 推荐展示方式 | 分层地图、关键竞品画像段落、竞争张力表、格局判断短段。 |
| 完成标准 | 读者能区分“谁和我方争同一任务/预算/用户”“谁只是替代或参考”“每个关键竞品强在哪里、限制在哪里”。 |

画像段落的核心不是句数，而是必须交代用户、任务、产品结构、增长入口、优势、限制和我方启示。信息充分时可以展开，证据不足时要标注边界。

### 3.3 用户痛点、场景与切换动因

| 契约项 | 要求 |
|---|---|
| 要回答的问题 | 谁在什么任务压力下选择这些产品？为什么采用、留存、付费或切换？哪些需求还没有被很好满足？ |
| 必须使用的证据 | 用户评价、客户案例、社区讨论、产品评论、官网场景、实际路径观察、行业用户变化资料。 |
| 必须完成的分析动作 | 区分买方/使用者/决策者/影响者/阻碍者，拆高价值任务，解释切换触发点和真实损失。 |
| 推荐展示方式 | 用户角色表、关键场景叙事、切换路径表、用户声音或推断画像。 |
| 完成标准 | 读者能看出“产品机会来自哪个具体人群和场景”，而不是只看到泛泛的目标用户分类。 |

场景数量不机械固定。标准深度通常应覆盖足以支撑判断的高价值场景；证据不足时，明确哪些是待验证假设。

### 3.4 商业模式、产品定位与增长逻辑

| 契约项 | 要求 |
|---|---|
| 要回答的问题 | 竞品为什么会长成现在这样？它们靠什么赚钱和增长？商业目标如何塑造产品边界和体验？ |
| 必须使用的证据 | 定价页、套餐权益、客户类型、销售入口、免费/试用策略、融资/财报/公开商业资料、增长入口。 |
| 必须完成的分析动作 | 对比价值主张、收入逻辑、增长入口、壁垒和定位差异；解释商业模式对体验的影响。 |
| 推荐展示方式 | 定位与商业逻辑对比表、商业模式影响分析段落、我方定位启示。 |
| 完成标准 | 读者能理解某些功能、权限、升级提示、销售触点和治理能力为什么存在，而不是只看到功能清单。 |

### 3.5 产品能力对比矩阵

| 契约项 | 要求 |
|---|---|
| 要回答的问题 | 竞品在关键能力上差异在哪里？哪些能力已经普遍具备，哪些能力存在明显空白或深度差异？ |
| 必须使用的证据 | 产品页、帮助文档、功能入口、更新日志、定价限制、实测路径、用户反馈。 |
| 必须完成的分析动作 | 建核心能力矩阵，标注能力支持深度、证据类型和关键差异。 |
| 推荐展示方式 | 能力对比矩阵、能力地图、关键差异段落。 |
| 完成标准 | 读者能快速看出竞品能力差异，以及这些差异对我方优先级意味着什么。 |

### 3.6 KANO / 功能取舍判断

| 契约项 | 要求 |
|---|---|
| 要回答的问题 | 哪些是基础型能力、期望型能力、兴奋型能力，哪些是过度建设或当前不该跟？ |
| 必须使用的证据 | 能力矩阵、用户痛点、核心路径、竞品策略、商业模式和我方目标用户。 |
| 必须完成的分析动作 | 按 KANO 或类似方式分类，识别标配、竞争焦点、差异化机会、暂不跟进能力。 |
| 推荐展示方式 | 功能取舍表、MVP 优先级表、能力空白与过度建设分析。 |
| 完成标准 | 读者能形成清晰的产品和体验优先级：必须补齐、应该强化、可以差异化、暂时不跟。 |

### 3.7 核心路径、流程机制与体验表达

| 契约项 | 要求 |
|---|---|
| 要回答的问题 | 哪些路径最影响理解、信任、效率、转化或留存？用户在流程里卡在哪里？竞品用什么机制推动下一步？ |
| 必须使用的证据 | 实测、官方 demo、帮助文档、第三方 walkthrough/截图/评测、可访问页面观察、通知/召回/运营活动或用户评论。 |
| 必须完成的分析动作 | 还原核心路径，记录关键阶段的用户阻力、竞品机制、亮点和摩擦，解释背后意图，转化为设计启示和功能机会。 |
| 推荐展示方式 | 路径图、阶段-阻力-机制表、任务矩阵、体验细节观察、设计启示段落。 |
| 完成标准 | 读者能知道“哪个阶段的哪个机制造成了体验差异”，并知道该学、该避开、该重设计，或该转化成什么功能。 |

### 3.8 机会优先级与风险反证

| 契约项 | 要求 |
|---|---|
| 要回答的问题 | 哪些机会真正值得优先做？哪些只是看起来合理？主要风险、反证和不成立条件是什么？ |
| 必须使用的证据 | 竞品证据、数据/行业证据、用户证据，以及相反证据或约束。 |
| 必须完成的分析动作 | 提炼重要机会，判断优先级，做交叉验证，写出不成立条件和风险。 |
| 推荐展示方式 | 机会优先级表、机会论证小节、风险与反证段落、P0/P1/P2。 |
| 完成标准 | 读者能判断机会的优先级和可信度，而不是只看到一串“可以做”的建议。 |

### 3.9 产品与体验决策

| 契约项 | 要求 |
|---|---|
| 要回答的问题 | 前面的事实最终转化成哪些产品方向、用户选择、功能策略、体验路径、表达方式和优先级？ |
| 必须使用的证据 | 前文已验证的关键事实、张力、路径观察、机会和风险。 |
| 必须完成的分析动作 | 把发现压缩为方向、用户、功能、体验、表达、增长和优先级判断，并说明判断之间的关系。 |
| 推荐展示方式 | 决策矩阵、综合判断段落。 |
| 完成标准 | 读者能把报告直接用于立项、改版、需求优先级或设计方案取舍。 |

### 3.10 行动建议与 Non-goals

| 契约项 | 要求 |
|---|---|
| 要回答的问题 | 接下来具体做什么？谁先做？如何验证是否有效？哪些方向明确不做？ |
| 必须使用的证据 | 产品与体验决策、机会优先级、风险反证、资源或阶段约束。 |
| 必须完成的分析动作 | 转成 P0/P1/P2 或时间窗动作，给出验证方式；新产品切入类报告说明 Non-goals。 |
| 推荐展示方式 | 行动优先级表、P0 建议展开、Non-goals 清单。 |
| 完成标准 | 建议不是口号，而是能进入后续产品、设计、研究或增长工作；读者也知道当前不该做什么。 |

### 3.11 来源、可信度与信息缺口

| 契约项 | 要求 |
|---|---|
| 要回答的问题 | 哪些结论可信？哪些只是推断？哪些信息没有采集、找不到、不可访问或互相冲突？ |
| 必须使用的证据 | 全部关键来源、采集状态、来源日期、证据类型和限制。 |
| 必须完成的分析动作 | 标注来源层级、覆盖状态、关键缺口、推断边界和下一步补证据方向。 |
| 推荐展示方式 | 来源覆盖表、关键数字可信度、重要推断清单、下一步补证据。 |
| 完成标准 | 读者知道这份报告能支撑什么判断，不能支撑什么判断，不会把初步推断当成事实。 |

## 4. 多品竞品分析报告模板（标准深度）

这是默认主报告的参考骨架。默认分析语境是多品竞品分析：建立竞品分层、横向能力差异、竞争张力、机会空白和我方取舍。它不是简报，而是一套结构完整、信息充分、以产品与体验决策为导向的标准深度报告。

优先采用“判断链版本”。只有在用户明确要求标准目录、组织内部需要可对照模板、或内容非常复杂需要显式分章时，才采用“模块目录版本”。

## {机会验证：哪些机会值得做，哪些可能是误判}

重要机会逐个论证：机会描述、为什么可能成立、为什么可能不成立、如何验证、我方产品/设计动作。可以补 P0/P1/P2 表，但表格不能替代论证。

## 产品与体验决策

- 免费版/试用版开放范围如何影响用户首次价值。

- 增长入口来自模板、协作邀请、生态集成、销售驱动，还是内容/社区。
- 商业目标如何影响权限、品牌露出、升级提示、协作限制和企业治理。

### 3.3 对我方的定位启示

用一段话说明我方应选择哪种增长路径：个人先行、团队协作扩张、企业销售驱动、套件生态绑定，还是垂直场景切入。

## 5. KANO / 功能取舍判断

不需要在正文解释 KANO 方法论，但要把能力转成产品和体验优先级。重点回答：哪些必须补齐，哪些值得强化，哪些可以形成差异化，哪些当前不该跟。

### 5.1 能力空白与过度建设

必须写出：

- 哪些是市场空白，但不一定适合我方。
- 哪些是用户真需求，但竞品只是浅层满足。
- 哪些是竞品看起来很强、但我方当前不该跟的能力。
- 哪些能力必须组合起来才有价值，不能单点建设。

先说明哪条路径最影响用户理解、信任、效率、转化或留存，以及竞品在这条路径上做了什么关键取舍。线上产品要进一步说明流程内机制：用户卡在哪里、竞品用什么机制推动下一步、这个机制能否转成我方可落地功能。

### 6.3 体验细节观察

按路径写具体细节，不要只写“体验好/体验差”：

- 首次价值：首屏文案、默认模板、空状态、示例数据如何帮助用户开始。
- 核心任务：步骤、信息层级、输入成本、反馈方式、失败恢复。
- 协作路径：邀请、评论、权限、通知、版本、交接。
- 表达方式：信息架构、文案语气、视觉层级、品牌感、动效和状态提示。

### 6.4 设计启示

用 1-3 段把路径观察转成设计判断：哪些体验要学习，哪些要避开，哪些需要结合我方用户重新设计。

## 5. 单品、赛道、快速请求适配规则

默认使用多品竞品分析报告。只有当用户明确给出单一产品、要求发现竞品/赛道分析，或明确要求快速简报时，才按以下规则调整。

开场不要求用户先选择报告类型或调研模式。对象清楚时默认按综合标准分析推进，并可用一句轻提示说明：“我会默认按综合竞品分析来跑；如果你更关心产品方向、功能取舍、体验路径、表达转化、商业增长或单点验证，我会相应加权。”只有当用户主动询问“应该怎么调研/选哪种分析”时，才展开解释不同目的类型。

### 4.1 单品对象

适合用户说“分析一下 X”“拆一下 X 的 AI 功能”“看一下这个产品为什么做得好”。

重点加深：

- **竞品格局与画像**：把“竞品格局”写成“实体画像 + 所属竞争位置”。说明它是什么、服务谁、靠什么增长、在赛道里占什么位置。
- **商业模式、产品定位与增长逻辑**：加深价值主张、收入逻辑、渠道、客户关系、核心资源和成本结构。
- **产品能力对比矩阵 + KANO / 功能取舍判断**：加深产品架构、核心能力、功能深度、能力类型和哪些能力承接了定位。
- **核心路径、流程机制与体验表达**：加深首次价值、核心任务、流程内机制、协作/转化路径和界面表达。
- **机会优先级与风险反证**：重点写“我们能学什么、不能照搬什么、它的优势是否可迁移”。

单品对象可按需加入：产品发展时间线、商业模式画布、产品架构图、战略信号表。

### 4.2 赛道/发现竞品对象

适合用户说“看一下这个赛道”“这个行业有哪些竞品”“帮我找竞品再分析”。

重点加深：

- **竞品格局与画像**：先做市场分层和候选竞品分类，必要时让用户确认重点对象；确认后进入多品竞品分析报告。
- **用户痛点、场景与切换动因**：加深人群细分、买方/使用者/决策者差异、高价值任务和切换路径。
- **商业模式、产品定位与增长逻辑**：加深主流商业模式、价格带、渠道和进入壁垒。
- **产品能力对比矩阵 + KANO / 功能取舍判断**：加深行业标配能力、新进入者必须补齐的能力和差异化能力。
- **机会优先级与风险反证**：重点判断切入窗口、替代方案、行业约束和进入风险。

赛道对象可按需加入：行业趋势摘要、权威报告数据、市场规模、进入壁垒、生态关系图。

### 4.3 快速请求

适合用户明确说“快速看一下”“先给我一个简版”“先做个判断简报”。

快速判断简报可以压缩为：

```markdown
# {对象/赛道} 竞品分析快速判断简报

## 0. 核心决策卡片
## 1. 竞品格局与关键对比
## 2. 核心发现
## 3. 产品与体验决策
## 4. 行动建议
## 5. 信息来源与缺口
```

快速版可以减少细节，但不能只给观点；必须保留关键事实、证据和推理链。
