# 完整报告目录脚手架

这是一份按需读取的固定目录参考，不是另一种默认报告模式。

只有在用户明确要求固定目录、组织内部需要逐章对照，或复杂报告确实需要显式章节脚手架时，才读取本文件。默认标准深度报告应使用 `report-templates.md` 的判断链版本。

即使读取本文件，也只把它作为覆盖检查和章节参考；最终仍须遵守 `report-templates.md` 的判断链、模块契约、标准深度门禁和展示规则，不得机械填表。

````markdown
# {对象/赛道} 全栈竞品分析报告

> 分析日期：{日期}
> 分析对象：{对象}
> 输出形态：多品竞品分析报告（标准深度）
> 判断方式：综合判断
> 聚焦问题：{如有}
> 目标拆解关键因素：{如有明确目标，列 4-7 个适配本行业和本目标的关键因素；无明确目标可省略}
> 证据覆盖：{覆盖 L1/L2/L4/L6 中几类；不足 3 类时标为初步判断版}

### 1.2 关键竞品画像

每个重要竞品不要只写一句定位。画像要完成本文件“竞品格局与画像”的模块契约，至少交代：

- 它主要服务谁，切入哪个核心任务。
- 它用什么产品结构承载价值，例如文档、数据库、项目、知识库、组件、工作流。
- 它的增长入口是什么，例如个人免费、团队协作、模板、生态、销售、套件绑定。
- 它最值得学习的地方是什么。
- 它最明显的限制是什么。

{解释产品逻辑、目标用户、核心能力、增长方式、优势、限制与我方启示。}

## 3. 商业模式、产品定位与增长逻辑

先解释产品为什么会长成现在的样子：它们面向谁收费，靠什么扩张，哪些功能是为了获客，哪些功能是为了转化或留存。

|---|---|---|---|---|

### 3.2 商业模式如何塑造产品体验

围绕以下问题说明：

- 免费版/试用版开放范围如何影响用户首次价值。
- 定价和套餐如何暗示目标用户是个人、团队、中小企业还是大型组织。
- 增长入口来自模板、协作邀请、生态集成、销售驱动，还是内容/社区。
- 商业目标如何影响权限、品牌露出、升级提示、协作限制和企业治理。

### 3.3 对我方的定位启示

用一段话说明我方应选择哪种增长路径：个人先行、团队协作扩张、企业销售驱动、套件生态绑定，还是垂直场景切入。

不需要在正文解释 KANO 方法论，但要把能力转成产品和体验优先级。重点回答：哪些必须补齐，哪些值得强化，哪些可以形成差异化，哪些当前不该跟。

| | 暂不跟进/过度建设 | | 当前不做或观察 |

### 5.1 能力空白与过度建设

必须写出：

- 哪些是市场空白，但不一定适合我方。
- 哪些是用户真需求，但竞品只是浅层满足。
- 哪些是竞品看起来很强、但我方当前不该跟的能力。
- 哪些能力必须组合起来才有价值，不能单点建设。

## 6. 核心路径、流程机制与体验表达

本节必须标注证据来源类型：实测、官方演示/官网、帮助文档、第三方截图/评测、推断。没有真实进入产品时，不要写成实测结论。

先说明哪条路径最影响用户理解、信任、效率、转化或留存，以及竞品在这条路径上做了什么关键取舍。线上产品要进一步说明流程内机制：用户卡在哪里、竞品用什么机制推动下一步、这个机制能否转成我方可落地功能。

### 6.1 核心路径拆解

| 协作/分享/权限 | | | | | | |
| 付费/升级 | | | | | | |

```text

```

### 6.2 流程内机制表

当目标涉及活跃、留存、转化、复购、创作、协作、关系沉淀或信任时，可以加入流程内机制表。它应该出现在竞争格局、用户场景和能力取舍之后，作为从路径观察走向机会验证的桥梁。表格不是为了填满模块，也不是报告主线；它只用于帮助读者看清“功能如何在流程里发挥作用”。

### 6.3 体验细节观察

按路径写具体细节，不要只写“体验好/体验差”：

- 首次价值：首屏文案、默认模板、空状态、示例数据如何帮助用户开始。
- 核心任务：步骤、信息层级、输入成本、反馈方式、失败恢复。
- 协作路径：邀请、评论、权限、通知、版本、交接。
- 表达方式：信息架构、文案语气、视觉层级、品牌感、动效和状态提示。
- 背后意图：降低学习成本、提高转化、推动协作、强化治理，还是制造高级感。

### 6.4 设计启示

用 1-3 段把路径观察转成设计判断：哪些体验要学习，哪些要避开，哪些需要结合我方用户重新设计。

## 7. 机会优先级与风险反证

重要机会至少用竞品、数据、用户中的可用证据验证。证据不足时明确标注。机会不只是“可以做”，还要判断优先级、不成立条件和风险。

### 7.3 风险与反证

专业报告不能只证明自己想要的结论，也要写：

- 哪些竞品能力可能被低估。
- 哪些用户痛点可能只是小众人群。
- 哪些方案会带来成本、合规、组织或体验风险。

## 10. 来源、可信度与信息缺口

说明这份报告当前能支撑哪些判断，哪些判断仍然需要补证据。不要把信息缺口写成客套话。标准深度报告至少覆盖 L1 官方产品源、L2 真实体验源、L4 权威行业报告/市场资料、L6 第三方口碑源中的 3 类；不足 3 类时，本报告必须标为“初步判断版”。

写信息缺口时必须区分：

- **本次未采集**：这次没有执行对应信息源采集，不能暗示资料不存在。
- **已检索但未获得有效材料**：已经搜索/访问，但没有找到可用证据，要简述检索方向。
- **可见范围受限**：资料可能存在，但受付费墙、登录、地域、权限或页面不可访问限制。
- **证据冲突**：不同来源结论不一致，需要保留冲突并说明影响。
````
